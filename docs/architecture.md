# Техническая архитектура (Technical Architecture) — MetaCMS Builder 🏗️

Этот документ содержит детальное описание архитектуры платформы **MetaCMS Builder**, ее модульной структуры, механизмов реактивного состояния и СУБД-трансляторов.

---

## 📂 Общая структура каталогов и Модульный дизайн

Кодовая база MetaCMS спроектирована по стандартам модульного монолита, разделяющего визуальную оболочку (Android/Compose), ядро бизнес-логики и скомпилированные веб-страницы (PWA):

```
├── /app                  # Клиентское Android-приложение (Jetpack Compose, MVVM)
│   ├── src/main/java/com/example
│   │   ├── data          # Локальный СУБД Room кэш, API-запросы Gemini
│   │   ├── ui            # Компоненты верстки, Темы M3 и Модуль перевода
│   │   └── viewmodel     # Мутабельные StateFlow-потоки и управление стейтом
├── /core                 # Компилируемые библиотеки ядра CMS на Dart/SaaS (Виртуальная модель)
├── /modules              # Набор полностью автономных функциональных плагинов
│   ├── Blog/             # Статьи, комментарии, категории
│   ├── CRM/              # Настройки лидов и сделок
│   ├── Shop/             # Карточки и корзины e-commerce
│   ├── AI/               # Обработчики интеллектуальных подсказок Gemini
│   └── Wiki/             # Документы и статьи баз знаний
├── /docs                 # Профессиональная техническая документация и ТЗ
├── /.github              # Конвейеры CI/CD для развертывания PWA и релизов APK
└── /docker               # Конфигурационные файлы контейнеризации Nginx
```

## ⚡ Описание слоев данных и реактивного состояния (MVVM Flow)

1. **База данных (Room SQLite):**
   - Все созданные воркспейсы пользователей сохраняются в локальном хранилище `CmsDatabase` через Room DAO.
   - Схемы колонок для каждого проекта сериализуются в JSON-строку и динамически распаковываются в коллекции объектов `SchemaField` при редактировании.
2. **ViewModel (`CmsViewModel.kt`):**
   - Использует `Kotlin StateFlow` для мгновенной трансляции изменений.
   - Каждое действие (добавление колонки, переключение темы или языка) является реактивным и немедленно вызывает перерисовку связанных Compose-компонентов.
3. **Синхронизация Git-контернеров:**
   - Каждому воркспейсу назначается целевой репозиторий `owner/repo` и ветка.
   - Операции Pull, Push и Release полностью имитируют реальный поток выполнения git-инструкций в консоли с выводом логов CI/CD.

## 🌐 Локализационный Стек (i18n Localization Engine)

Локализация реализована как статическая словарная карта в файле `Localization.kt` с быстрым хэш-поиском по ключам UI на лету:
```kotlin
fun t(key: String): String {
    val lang = _selectedLanguage.value
    val actualLang = if (lang == "System") {
        com.example.ui.locale.Localization.mapSystemLanguage(java.util.Locale.getDefault())
    } else {
        lang
    }
    return com.example.ui.locale.Localization.translate(key, actualLang)
}
```
Такой подход исключает асинхронную задержку при загрузке строк и минимизирует вероятность показа пустых экранов.
